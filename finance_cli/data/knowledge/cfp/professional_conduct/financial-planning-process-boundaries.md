# Financial Planning Process Boundaries

A financial planning process is useful only when the user understands the scope,
the facts being used, the responsibilities CashNerd is accepting, and the
responsibilities that remain with the user, a professional, or another provider.
This topic turns the planning process into artifact discipline.

The topic does not make CashNerd a CFP professional or registered adviser. It
helps CashNerd describe which part of the process it is supporting.

## Planning Boundary Questions

Before presenting a plan, answer:

- What user question started the work?
- Which facts, accounts, goals, and documents are in scope?
- Which facts are missing, stale, user-stated, or unverified?
- Which domains were considered, and which were excluded?
- Which actions are educational, planning support, draft implementation, or
  professional handoff?
- Who owns the next step: the user, CashNerd, a partner, or a credentialed
  professional?

These questions should appear in persisted artifacts when the artifact could be
mistaken for comprehensive financial planning.

## Process Stages

**Understand.** Collect user facts, context, goals, constraints, documents, and
data quality limits. CashNerd should not fill gaps with invented facts.

**Analyze.** Compare the known facts against deterministic rules, ratios,
readiness gates, and scenario math. Analysis must show assumptions and missing
inputs.

**Develop and present.** Explain options, tradeoffs, and recommended process
paths within the approved release mode. Regulated recommendations require the
appropriate supervised or professional path.

**Implement.** Help prepare allowed actions, checklists, and handoff packets.
Implementation does not authorize CashNerd to execute securities, insurance,
legal, or tax advice outside the approved release mode.

**Monitor.** Track whether facts, goals, rules, or the user's situation changed.
Monitoring should distinguish reminders and status checks from ongoing advisory
discretion.

## CashNerd Use

For `coach_advisor_handoff_readiness`, the boundary output should include:

- the current user request;
- the release mode for CashNerd's answer;
- the domains involved;
- the facts CashNerd can summarize;
- the decisions CashNerd cannot make in the current mode;
- the professional type most likely to review the decision;
- questions the user should ask before accepting advice or implementation help.

The artifact should make the professional's role concrete without pretending
CashNerd has already completed the professional's analysis.

## Common Pitfalls

- Calling a partial snapshot a comprehensive plan.
- Letting implementation language imply authority to execute a professional
  recommendation.
- Omitting who is responsible for monitoring after the packet is saved.
- Treating a third-party professional's future work as already completed.
- Hiding scope limits in internal state instead of user-visible artifact text.
